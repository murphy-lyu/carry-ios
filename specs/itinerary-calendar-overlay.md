# Itinerary Calendar Overlay（日历事件叠加层 — 行程规划里显示系统日历事件）

> **Status: Draft — 待确认，未实现。** 本 spec 定义数据来源、隐私边界、交互与范围，确认后再编码。
>
> **一句话**：把用户**自己勾选的**系统日历里、落在行程日期内的事件，作为**只读叠加层**显示在行程规划的每天时间轴里——**永不写进行程数据、永不进任何分享/导出/备份**。借 Tripsy 的「当天日历事件入列表」，但用 Carry 的克制 + 一条不可妥协的隐私红线做取舍。

## 动机

出门旅行那几天，系统日历里的事件**常与行程相关**：订的餐厅 / 演出 / Tour、加进日历的航班、尤其是**当地节假日**（影响开门、人流、交通、价格）。Carry 已有 EventKit（现在就往日历**写**行程/打包提醒），**读**事件是顺水推舟。把这些事件就地铺进行程，能在「这天要干嘛」给一个统一视图，且零手动录入。

但**不能照搬 Tripsy 的全量塞入**——用户旅行期的日历也有非行程事件（工作会议、看医生、私人约会），全量既吵又敏感，且 Carry 能分享/导出行程，混入日历会泄露隐私。所以本功能的核心不是「拉数据」，而是**「在隐私红线内拉对的那一刀」**。

## 🔴 隐私红线（本功能的设计核心，不可妥协）

1. **只读**：仅从 EventKit **读取**，绝不修改/删除用户日历事件。
2. **不入模型**：日历事件**永不**写进 SwiftData / `TripBundle` / `ItineraryDay`。它是**视图层的实时查询**，每次渲染时从 EventKit 取、用完即弃。
3. **永不分享 / 导出 / 备份**：`.carrytrip`（`makeItineraryShareFile`）、签证 PDF（`ItineraryPDFRenderer`）、分享海报（`TripSharePoster`）、`DataBackupManager` **一律不含**日历事件。
   - **由构造保证**：因为事件从不进 model，这些导出路径天然取不到它——这是最强的保证（不是「记得过滤」，而是「根本不在数据里」）。落地时仍要**显式测试**：含日历叠加的行程，导出/分享/备份产物里无任何日历事件。
4. **opt-in + 用户选日历**：默认**关**；开启后用户**自己勾选要显示哪些日历**（像系统日历的日历开关）。我们不替用户猜「哪些算行程相关」——把选择权交给他，从根上解决隐私+噪音。
5. **过滤掉 Carry 自己写的事件**：Carry 通过日历同步往日历写过「行程/打包提醒」；若用户勾选的日历恰好包含它们，必须**识别并排除**，避免「Carry 写的提醒又被 Carry 当日历事件显示回来」的回环。识别方式见下（事件 URL / notes 标记或专用日历）。

## 核心设计原则

1. **叠加层，不是行程数据**：视觉与语义都和停靠点/交通/住宿**明确区分**——它来自「你的日历」，不是「你规划的行程」。
2. **克制**：默认关、用户选源；不做编辑、不做提醒、不做双向同步。只「读 + 显示」。
3. **顺平台**：用 EventKit 原生能力 + 系统日历颜色；权限走 iOS 17+ `requestFullAccessToEvents`。
4. **节假日/全天事件价值最高、最不敏感**——是本功能的甜区；定时私人事件由用户自行决定是否勾选其所在日历。

## 数据来源与查询

- **框架**：EventKit，只读。复用现有 `EventStore`（日历同步已建立的访问；注意权限升级，见下）。
- **范围**：行程日期区间（`departureDate` → `departureDate + days`）。**无日期（dateless）行程不支持**（无区间可查）——该行程不显示叠加层。
- **过滤**：
  - 仅用户**勾选的日历**（存 `selectedCalendarIdentifiers: [String]` 于 UserDefaults）。
  - 排除 **Carry 自己写入的事件**（见隐私红线 5）。
  - 可选：排除「已拒绝/取消」状态事件。
- **权限**：读事件需 EventKit **完整访问**（iOS 17 区分 write-only / full）。现有日历同步若只申请了写权限，需在开启本功能时申请 `requestFullAccessToEvents`；拒绝则功能不可用并给出引导。

## 渲染（行程时间轴里）

- **位置**：并入每天时间轴，按时间排序——全天事件置于当天顶部；定时事件按 start time 落在停靠点/交通序列里（与现有 timeline 时间就位逻辑一致）。
- **视觉（与行程数据区分）**：参考 Tripsy——**左侧一道竖条用该事件所属日历的颜色** + 事件标题 + 时间；整体更轻（recessive），**不挂 rail 圆 marker**（它不是行程节点）。明暗自适应。
- **多天全天事件**（如假期块）：在其覆盖的每天都显示（呼应住宿跨天三态的处理）。
- **交互**：只读。点击**可选**唤起系统日历查看该事件（`EKEvent` deep link / 打开日历 App），但**不在 Carry 内编辑**。不参与重排、不可拖动。
- **空/降级**：未授权 / 未选日历 / 该天无事件 → 不显示任何叠加行（静默，无空态噪音）。

## 设置入口

- 放进现有「日历同步」设置区（`CalendarSettingsView`，spec: `calendar-sync.md`）。
- 新增：**「在行程里显示日历事件」开关**（默认关）+ 开启后的**日历多选**列表（系统日历名 + 颜色 + 勾选）。
- 文案说明：明确「只读、不会分享给同行者、不写入行程」，打消隐私顾虑。

## 埋点

- `calendarOverlayEnabled` / `calendarOverlayDisabled`：衡量开启率。
- `calendarOverlayCalendarsSelected`（带数量）：衡量用户选了几个日历。
- 错误类：`calendarOverlayAccessDenied`（读权限被拒）→ 纳入 `errorEvents`。

## 本地化

- 新结构化 key（×9 语言、含显式 en）：`settings.calendar.overlay.title`、`settings.calendar.overlay.footer`（含隐私说明）、`settings.calendar.overlay.pick`、`itinerary.calendar.section`（时间轴里「来自日历」轻标签，若需要）等——最终清单实现时定。

## 明确不做（范围边界 · 克制）

- ❌ 全量倒入所有日历（噪音 + 隐私）——必须用户选源。
- ❌ 把日历事件写进行程 / 分享 / 导出 / 备份（隐私红线）。
- ❌ 在 Carry 内编辑 / 创建 / 删除日历事件（只读）。
- ❌ 双向同步、提醒、把日历事件转成停靠点。
- ❌ 无日期行程的叠加层（无区间）。
- ❌ 自动判断「哪些事件算行程相关」——交给用户选日历，不做启发式猜测。

## 实现阶段（确认后）

1. **读取层**：EventKit 完整访问申请 + 行程区间查询 + 选中日历过滤 + 排除 Carry 自写事件。纯数据，先不接 UI。
2. **设置**：开关 + 日历多选（存 identifiers）+ 隐私说明文案。
3. **时间轴渲染**：叠加行（左色条 + 标题 + 时间，轻量、无 rail marker），按时间就位；多天全天事件跨天显示。
4. **🔴 隔离验证**：含日历叠加的行程，`.carrytrip` / PDF / 海报 / 备份产物里**零日历事件**——显式测试这条红线。
5. **埋点 + 本地化（9 语言）+ 编译 + 真机验收**（含权限拒绝/未选日历/跨天全天事件/Carry 自写事件被正确排除）。

## 开放问题（待你拍板）

1. **默认范围**：开启后默认**不选任何日历**（最安全，用户逐个开）？还是默认**建议勾选「节假日」类日历**（更省事，但要能可靠识别节日日历——iOS 不一定有干净标识）？我倾向**默认不选、用户自己开**（红线最稳）。
2. **是否允许点击事件唤起系统日历**：增便利但引一次 App 跳出。我倾向**可点、唤起系统日历查看**（只读，不在 Carry 编辑）。
3. **定时 vs 仅全天**：先只做**全天事件（含节假日）**这个甜区，定时事件留 Phase 2？还是一开始就含定时（用户选的日历里全显示）？我倾向**一开始就按「用户选的日历」全显示**（全天+定时），因为隐私已由「用户选源」兜住，再按类型砍反而割裂。
