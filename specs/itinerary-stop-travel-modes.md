# Itinerary Stop Travel Modes（停靠点详情：多交通方式到下一站时长）

> **Status: Draft — 待确认，未实现。** 本 spec 定义功能与交互，确认后再编码。
>
> **依赖：** 建立在 `itinerary-stop-detail.md`（只读详情 + 路程模块）已落地之上——本功能是把详情底部「路程模块」从「单一直线距离」升级为「多交通方式时长」。与 `itinerary-stop-navigation.md`（Get Directions）并存。

## 动机

停靠点详情底部现在只显示「到下一站 N 公里」（直线距离，瞬时、离线）。用户参考 Tripsy 的「旅行时间」——希望看到**到下一站的多种交通方式时长**（步行 / 驾车 / 公交 …），默认驾车，一眼比较「走过去 vs 打车 vs 坐公交」分别多久，辅助安排当天节奏。

这是真实规划价值：直线距离回答不了「这段路实际多久」。

## 🔴 硬约束（先讲清楚，决定范围）

1. **Apple `MKDirections` 只支持 `.automobile` / `.walking` / `.transit`，不支持「骑行」。** 因此 Tripsy 那个「步行·驾车·公交·骑行」四态矩阵，**用 Apple 原生 API 做不全**——它的骑行多半接了 Google/其它源。**本 spec 首版只做三态（驾车/步行/公交）**；骑行作为「未来接非 Apple 源」的待办，不在首版。
2. **联网路由**：`MKDirections` 是异步网络请求，有加载/失败/离线态。现在「直线距离」是瞬时离线的；升级后必须优雅处理「等网返回」与「无网/失败」。
3. **公交时长不一定可得**：`.transit` 的 ETA 依赖该地区交通数据。中国大陆 Apple 地图走高德，公交 ETA 可能返回、也可能不返回——**必须能优雅缺省**（该态显「—」或隐藏，不报错、不空等）。

## 核心设计原则

1. **克制**：三态一行、一眼比较，默认驾车高亮。不堆 Tripsy 的完整矩阵 + 路线段 + 起讫站。
2. **直线距离作离线兜底**：联网未回 / 失败 / 无网 / 无下一站 → 回退现在的「到下一站 N 公里」，绝不空白或卡死。
3. **不每次开详情都重算**：路由结果按「停靠点对（from→to）坐标」缓存；坐标没变就用缓存，再开瞬时出。
4. **顺平台**：用 `MKDirections.calculateETA`（只要时长/距离、不取完整路线，轻量），不自造路由；坐标用地图同款（不额外做坐标系转换——与现有地图针/直线距离同源，中国大陆下 MapKit 已是高德口径）。
5. **不破坏导航**：Get Directions（`MapNavigationService`，调起外部地图）保持；本功能只是在它上方加「时长参考」。

## 交互 / UI

详情底部「路程模块」（现：Get Directions + 「到下一站 N 公里」）升级为：

```
┌─────────────────────────────────────────┐
│  🚗 30 分钟   🚶 4 小时   🚆 35 分钟       │  ← 三态时长行，驾车默认高亮
│  ─────────────────────────────────────   │
│  ➤ Get Directions                      › │  ← 保持
└─────────────────────────────────────────┘
```

- **三态时长行**：`.automobile` / `.walking` / `.transit`，各显「图标 + 时长」。**默认驾车高亮**（视觉上比另两态略实，表「当前参考」）。
- **某态不可得**（公交无数据 / 该态路由失败）→ 显「—」或灰掉该态，不空等、不报错。
- **加载态**：路由未回时，时长行显占位（骨架/「…」）或先显直线距离；回来后填入。
- **离线 / 全失败**：整行回退为现在的「到下一站 N 公里」直线距离（与升级前一致）。
- **无下一站（末站）/ 无坐标**：不显时长行（按现状只给 Get Directions / 无坐标提示）。
- 高亮态是否**改变 Get Directions 调起的交通方式**？→ **开放问题**（见下）。首版倾向「纯展示比较，不联动」，Get Directions 行为不变。

## 数据 / 技术

- **请求**：对「本停靠点坐标 → 下一停靠点坐标」，分别用 `.automobile` / `.walking` / `.transit` 发 `MKDirections.calculateETA`，取 `expectedTravelTime`（+ `distance` 备用）。
- **时长格式化**：用 `DateComponentsFormatter`（.abbreviated，随地区本地化「分钟/小时」）；不手拼。
- **缓存**：内存缓存，键 = 「from 坐标 + to 坐标（量化到米级）+ mode」→ 时长。详情出现时若命中缓存即瞬时显示，否则发请求并回填缓存。坐标变化（编辑了地点）自然 miss → 重算。（首版内存即可；是否持久化见开放问题。）
- **并发 / 取消**：三态并发；详情关闭 / 切换地点时取消在途请求（用 `Task` + `.task(id:)` 生命周期绑定，避免回调打到已关闭的视图）。
- **节流**：同一停靠点对短时间内不重复发（缓存即天然节流）。
- **触发点**：`StopDetailView` 出现时（`.task`）对「to 下一站」发起；末站 / 无坐标不发。

## 范围 / 不做（首版）

- **不做骑行**（Apple `MKDirections` 不支持；待未来接非 Apple 源单列）。
- **不做** Tripsy 的「起讫站 + 路线段 + 到达时刻」完整面板——只做「三态时长一行 + Get Directions」。
- **不在 App 内画路线 / turn-by-turn**：导航仍调起外部地图（`MapNavigationService`）。
- **不做实时路况叠加**（`calculateETA` 自带的 ETA 即可，不另接路况源）。
- 不对没有下一站 / 没有坐标的停靠点计算。

## 开放问题（待拍板）

1. **高亮态是否联动 Get Directions 的交通方式**？纯展示比较（首版倾向）vs 选中某态 → Get Directions 调起对应方式（需确认 `MapNavigationService` 各 App 是否支持指定 mode 的 URL scheme）。
2. **公交（`.transit`）在中国大陆高德下到底返不返回 ETA**？需真机/模拟器实测——若基本不返回，首版是否干脆只做「驾车 / 步行」两态，避免常驻「公交 —」的半残观感。
3. **缓存持久化**？内存（关 App 即清）够用，还是落 SwiftData/轻存储跨会话（路由结果会过期、价值有限，倾向只内存）。
4. **加载态形态**：骨架占位 vs 先显直线距离再替换为时长（倾向后者——先给离线兜底值，回来再升级，无空窗）。
5. **是否埋点**：看了哪个 mode / 从详情发起导航——首版倾向不埋，避免过度。

## 待办 / 分阶段（确认后）

1. 实测 `.transit` 在大陆高德下的可得性（决定首版是 2 态还是 3 态）。
2. `TravelTimeService`（新）：`calculateETA(from:to:mode:)` + 内存缓存 + 并发/取消；坐标用地图同款。
3. `StopDetailView` 路程模块升级：三态时长行（默认驾车高亮）+ 加载/缺省/离线兜底；保留 Get Directions。
4. 文案：交通方式名（驾车/步行/公交）、时长格式、「—」缺省、「到下一站」等 × 9 语言（结构化 key 显式写 en）。
5. 与 `itinerary-stop-detail.md` 收敛（路程模块小节标注「已升级为多交通方式时长」）。
6. 真机验：联网/弱网/无网三种下的加载与兜底；中国大陆高德口径下三态可得性与坐标正确性。
