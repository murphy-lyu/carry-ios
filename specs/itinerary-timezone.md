# 行程时区系统化（Itinerary Time Zones）

**Status:** Draft — 待用户确认后实现
**提出:** 2026-06-21（对应长期 TODO「全 App 时区未系统化」）
**关联:** itinerary-transport-lodging.md / itinerary-car-rental.md / itinerary-airport-search.md / itinerary-flight-search-first.md

---

## 1. 问题与现状（已核对代码）

行程是跨时区的，但 Carry 现在的时间处理是**半套且不一致**：

- 所有时间存为**裸的「当天午夜起分钟数」+ `dayOrder`**（`departLocalMinutes` / `arriveLocalMinutes` / `checkInMinutes` / `checkOutMinutes` / `plannedStartMinutes`…），**不带时区**。
- **只有交通段（含租车，已并入 `TransportSegment`）带真时区**：`fromTimeZoneId` / `toTimeZoneId`（来自机场库）。截图里 picker 的「时区: Asia/Shanghai」按钮就是 transport 编辑页特有的。
- **地点（`Stop`）与住宿（`LodgingStay`）没有时区**：`NotificationManager` 里 `absoluteDate(… tzId: "")` 回退到**设备当前时区**。

### 🔴 由此产生的真 bug
人在上海、巴黎酒店 11:00 退房提醒 → 现在按**设备时区（上海）11:00** 触发，差掉巴黎与上海的时差。地点提醒、每日提醒同理（`tzId: ""`）。多时区行程下，地点/住宿的通知时刻是错的。

---

## 2. 核心原则

1. **行程时间 = 活动所在地点的「当地墙上时间」。** 用户在巴黎看到的 15:00 入住，就是巴黎的 15:00。这是不可动摇的心智模型。
2. **显意义，藏机制**（Carry 克制气质）：对用户呈现「当地时间 + 仅跨区时提示」，把「时区」这个烧脑概念**自动化、藏起来**，不怼到每次设时间的操作上。
3. **时区是地点的属性，不是用户该做的选择**：能从地点坐标自动推导，就不该让用户手选。手动切换只作**罕见兜底**。
4. **存「墙上时间 + 时区」，算「绝对时刻」**：通知/倒计时/"现在"线/跨段时长一律用绝对时刻（对齐已有航班的做法、以及 CostBearing「真相 + 派生」范式）。

---

## 3. 决策（均按最佳解落定）

| # | 决策 | 选定方案 |
|---|---|---|
| D1 | 单时区行程是否显示时区字样 | **完全不显**（零噪音） |
| D2 | 多时区行程的显示策略 | **全局切换**：整个行程出现 ≥2 个不同时区时，**每个有时间的条目**都带一个克制的时区小标；单时区则全程不带。比"只在边界标"更不易迷失 |
| D3 | 跨区当天的强调 | 时区**首次改变的那一天**，在 Day 头/该条目给一个稍明确的标记（如「自此 · 巴黎时间 GMT+1」），其余条目用轻量小标 |
| D4 | 提示形态 | 优先**城市/地名 + 偏移**的轻量小标（如「巴黎 GMT+1」或仅 `GMT+1`，secondary 灰）；不喧宾夺主 |
| D5 | 是否保留手动切换器 | **保留，但降级**为安静可点的小标签（仅兜底/自动推导失败时用）；现在 transport 那个醒目「时区」按钮一并降级、纳入统一组件 |
| D6 | 浮动时间（Apple 日历那种"到哪都 9:00"） | **不采用**。地点绑定的预订（退房/取还车/订位）必须锚定到地点时区，否则通知在错误时刻触发 |
| D7 | 默认时区来源 | 活动**地点的当地时区**（坐标推导）→ 取不到则行程主时区 → 设备时区 |

## 4. 模型设计

### 4.1 存储
- 给 `Stop`、`LodgingStay` 各加 `timeZoneId: String?`（IANA，如 `Europe/Paris`）。`TransportSegment` 已有 `from/toTimeZoneId`，纳入同一套语义。
- **可选类型**保持向后兼容；**发布前不升 `currentBackupVersion`**（按 CLAUDE.md 备份规则）。SwiftData 走 versioned schema 轻量迁移（新增可选字段）。
- 旧数据 `timeZoneId == nil` → 读取时按**回退链**解析（见 4.3），不阻塞用户。

### 4.2 时区推导管线（写入即捕获）
- 设定/更换地点时，从结果捕获 IANA 时区并存下：
  - 地点搜索（MapKit）：`CLPlacemark.timeZone`（`MKMapItem.timeZone` / 反查坐标）。
  - 机场：机场库已带 IANA。
  - 住宿/租车地点：同地点搜索路径。
- 与已有「电话/地址随 placemark 回填」同一处捕获，避免散落（参照 itinerary-transport-lodging.md 的回填模式）。

### 4.3 回退链（推导不到时）
活动地点解析不出时区时（手输无坐标的地点、只填了公司名的租车…）：
`同日前一个有时区的活动 → 行程主时区 → 设备时区`。
- **行程主时区** = 全行程有时间活动中出现最多的时区（并列时取最早出现的）；也用于 D2 的「单/多时区」判定。

### 4.4 绝对时刻
`absolute = (行程出发日 + dayOrder 推出的日期, minutes, timeZoneId)` 经 `Calendar(timeZone:)` 求出。统一一个 helper，替换 `NotificationManager` 里散落的 `tzId: ""`。

## 5. UX 规则

### 5.1 显示（D1/D2/D3/D4）
- **判定**：`distinctZones(trip).count <= 1` → 单时区模式；否则多时区模式。
- **单时区**：时间就显「18:30」，**全程无任何时区字样**（含 picker——降级隐藏切换器，见 5.2）。
- **多时区**：每个有时间的条目时间旁加**轻量时区小标**（secondary 灰，城市/`GMT±N`）；时区**首次切换的那天**给稍明确的边界标记。
- 参照系是「条目自己的时区 vs 行程主时区/前一条」，**不**依赖"设备当前时区"——因为旅途中设备时区本身在变，不能当稳定参照。

### 5.2 选择/编辑（D5）
- Picker 默认**只选时间**，时区由地点自动带出，**不显切换器**（单时区）。
- 多时区或自动推导失败时，picker 底部出现**安静的、可点的时区小标签**（非现在那种醒目实心按钮）→ 点开才进时区选择。像 Apple 日历的时区覆盖：藏着、够用、不挡路。
- 现在 transport 编辑页那个醒目「时区: Asia/Shanghai」按钮 → 收进这套统一组件、同样降级。

### 5.3 通知（修 bug）
- 退房/取还车/地点/每日提醒一律用**活动自己的时区**算绝对时刻（替换 `tzId: ""`）。航班维持 from/to 区不变。
- 文案口吻不变（本会话刚打磨过：只提还车/退房、航班 vs 非航班两套口吻等）。

### 5.4 按天分组（低风险，明确不改语义）
- **「在第几天」仍由用户指定的 `dayOrder` 决定，时区不改变分天归属**——时区只影响「绝对时刻」（通知/"现在"线/跨段时长）与显示提示。
- 同一天内的列表顺序仍按 `sortOrder` / 当地分钟数（直观）；绝对时刻仅用于时间机器（通知、now、倒计时）。

## 6. 全链路同步检查（改动必须覆盖，漏一处即丢/算错）
参照 CostBearing / 附件的「四处同步」纪律：
- ① **DataBackupManager**：`timeZoneId` 进备份/还原（可选字段、向后兼容）。
- ② **duplicateTrip**：深拷贝带上 `timeZoneId`。
- ③ **写入漏斗**：设定地点/时间时就地捕获并写时区，禁止 View 里裸写 `@Model` 时间字段而漏时区。
- ④ **TripStore / 通知 / 时间轴 / 详情卡 / 表单 chip**：统一走绝对时刻 helper 与显示规则。
- ⑤ 迁移：新增可选字段 + 旧数据读时回退链。

---

## 7. 分期落地

- **Phase 1 · 正确性（不可见，先做）**：模型加字段 + 推导管线 + 绝对时刻 helper + 通知改用活动时区。**对单时区用户零可见变化，但修掉跨时区通知 bug。** 这是真正必须的部分。
- **Phase 2 · 多时区显示**：D2/D3/D4 的时区小标 + 跨区边界标记（仅多时区行程出现）。
- **Phase 3 · 兜底切换器**：D5 的安静切换器，统一并降级现有 transport 时区按钮。

---

## 8. 验收要点（实现后）
- 单时区行程（纯国内/纯日本）：界面**无任何**时区字样；通知准。
- 跨时区行程（上海→巴黎→纽约）：每条带正确当地时间 + 轻量时区小标；巴黎退房提醒在**巴黎时间**触发（修 bug）；跨区当天有边界标记。
- 自动推导失败的地点：回退链生效、不报错；可手动经安静切换器纠正。
- 备份/还原、复制行程：时区不丢。
