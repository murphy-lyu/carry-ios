# 通知配额管理（iOS 64 条挂起上限）

Status: In Progress（2026-06-21 立项 + 实现）
关联：`notification-center.md`（通知中心架构）、`itinerary-timezone.md`（事件时区）

## 背景与问题

iOS 对每个 App 的**挂起本地通知**有硬上限：**最多 64 条**。超出后系统**只保留触发时间最近的 64 条，其余静默丢弃**，App 收不到任何错误回调。

Carry 的通知全部由 `NotificationManager` 调度（经核查全仓唯一调度入口；经期提醒只是打包推荐、不排通知），命名空间 `carry.trip.{tripId}.…`。单个多日多段行程可排出的通知量：

- 出发提醒：启用档位数（最多 4：当天/前1/前3/前1周）
- 打包提醒：1
- 交通出发：交通段数 × 提前量档数
- 退房：住宿条数
- 每日摘要：有安排的天数

一个 10 天、多航段的行程轻松排到 20–30 条；两三个这样的行程叠加即破 64。**原实现各行程独立调度、无全局视野**，越界后由系统不透明地丢弃远端通知，且**丢弃后不会自动补回**（无周期性重排）。后果：用户的远端行程通知可能悄悄排不上，且无任何提示。

## 设计目标

1. **App 掌控取舍，而非交给系统静默丢弃**——由我们决定排哪些，确定、可解释、可演进。
2. **近的优先**——触发时间越近的通知行动价值越高（即将发生的事最该提醒）；远端通知留到它临近、配额腾出时再进窗口。
3. **滚动补位**——随时间推移近端通知逐条触发释放名额、远端行程逐渐临近，需周期性重排把新进入「最近 N 条」窗口的通知排上。
4. **留安全余量**——不贴满 64，给 DEBUG 测试通知 / 未来可能新增的其它本地通知留缓冲。

## 方案

### 1. 全局预算调度（取代「各行程独立调度」）

新增唯一入口 `NotificationManager.reschedule(trips:)`：

1. **主线程**遍历所有非无日期行程，把每类通知构建成值类型快照 `Candidate`（id / fireDate / 文案 / 时区 / 是否 60s 兜底）。**不在此处调用 `add()`**——读取 `@Model` 全部留在主线程，回调里只用值类型。
2. 进 `getPendingNotificationRequests` 回调（拿 pending 快照）：
   - `foreign` = 非 `carry.trip.` 前缀的挂起数（DEBUG 测试通知 / 未来其它本地通知）
   - **预算** `budget = max(0, 64 − foreign − safetyMargin)`（`safetyMargin = 4`）
   - 候选**按 fireDate 升序**取前 `budget` 条为 `chosen`
   - **删除**「`carry.trip.` 前缀匹配但不在 `chosen`」的陈旧挂起；**新增** `chosen`（同 id 自动替换）
   - DEBUG 下若发生裁剪，打日志记录丢弃数

### 2. 竞态安全（沿用并强化重排修复）

通知 id 确定性，`add()` 同 id 替换。删除集 = 前缀匹配 − `chosen`，与新增集 `chosen` **天然不相交** → 无论 `getPending` 回调早晚都不会误删刚排的通知。模型读取在主线程、回调只碰值类型/字符串 → 无 off-main `@Model` 访问。

### 3. 滚动补位触发

- **冷启动**：`CarryApp` `onAppear` → `store.rescheduleAllTrips()`
- **回前台**：`willEnterForeground`（不在冷启动重复触发）→ 同上
- 既有的**逐变更**重排（建/改/删行程、增删事件、改设置、打包进度门控）全部改走全局入口

每次重排都重新排序全部候选、填满最近 `budget` 条 → 远端通知随临近 + 用户开 App 自然补位。

### 4. 调用链收口

TripStore 所有 `NotificationManager.scheduleReminders(for: trip)`（单行程）改为 `refreshNotifications()`（= `NotificationManager.reschedule(trips: trips)`，全局）。单行程语义已不成立——卡全局预算必须有全局视野。删除行程仍先 `cancelReminders(forTripId:)` 即时清理，再 `refreshNotifications()` 补位释放的名额。

## 决策与非目标（Minimal Effective Set）

- **近端优先，不做按类配额**：长行程的靠后每日摘要因 fireDate 远，自然让位给更近行程的出发提醒，时间维度已给出合理公平性；额外的「每类保底名额」增加复杂度而收益不明，明确**不做**。
- **不做防抖**：逐变更全局重排在常见行程量（个位数）下开销可忽略；打包进度重排默认关。若未来出现高频抖动再议。
- **残余不可避免边界**：当 >budget 条通知**全部**落在用户长期不开 App 的近期窗口（如未来 3 天排 80 条且 3 天不开 App），仍只触发最近 budget 条——这是 64 上限的物理边界，任何方案都无法绕过，仅在文档登记。
