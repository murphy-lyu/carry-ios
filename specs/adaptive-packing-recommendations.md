# Adaptive Packing Recommendations（基于用户习惯的自适应场景推荐）

> **Status: Draft（待讨论，未实现）** — 本 spec 仅描述方向与设计，需评审确认后再进入实现。**明确定位为"上线后"功能**：需要真实用户数据才能调阈值，上线前保持现有静态 `sceneItemMap` 不动。
>
> **一句话**：把当前写死的"场景 → 物品"推荐表，从"全体用户共用一套、一成不变"升级为"以静态表为冷启动基线、随每个用户的打包历史在本地悄悄自适应"。
>
> **来源**：同事内测反馈"现有场景推荐一成不变" + PM 进一步提出"能否记录用户出行习惯(去某地必带某物 / 后来又不带了)动态更新推荐"。

## 动机

现状：`SceneItemMap.swift` 里 `sceneItemMap: [String: [SceneItem]]` 是常量，所有用户共用同一套推荐。问题：
- 默认清单不可能对所有人都"刚好"——有人去热带必带某物、有人从不带；
- 用户每次都要重复地删掉不要的、加上自己惯用的，App 学不到任何东西。

目标：让推荐**跟着用户长**。去某类地方/选某个场景时，逐渐推得更准；用户戒掉的习惯也能被"忘掉"。这是相对其他打包 App（清单永远是死的）的差异化点。

## 核心设计原则

1. **静态表是冷启动基线，永不被删空**。`sceneItemMap` 继续作为 day-one 默认（新用户、新场景、新目的地都立刻可用）。个性化只是**在其之上叠一层调整**，不替代它。
2. **纯本地、零后端、无 ML**。本质是"频率记账 + 保守启发式"，契合隐私优先 / 大陆合规 / 当前无服务器。以后接 iCloud 同步时，习惯数据跟随用户自己的 CloudKit 数据走。
3. **避免过度抽象**（项目代码风格）。不引入推荐模型；就是 `(场景, 物品) → 计数`。
4. **隐形，不张扬**。不做"我们发现你总带 X"这类带监视感的提示。推荐**悄悄变好**即可；需要露出时走已有的"顺手考虑一下"(`SurpriseItemMap`)，保留用户最终决定权。
5. **保守 + 可逆**。少量样本不改动；核心基础项（护照/充电器等）永不被个性化移除；近期行为权重高于久远行为（习惯会变）。

## 数据模型

### 新增 SwiftData model：`PackingHabit`
```swift
@Model
final class PackingHabit {
    var contextKey: String      // 主键维度，见下「上下文粒度」
    var itemName: String        // 物品名（与 itemNameAliases 对齐，重命名须留 alias）
    var keptCount: Int          // 推荐了 + 用户留下
    var removedCount: Int       // 推荐了 + 用户删掉
    var manualAddCount: Int     // 未推荐但用户手动加
    var lastSeen: Date          // 用于时间衰减
    // contextKey + itemName 唯一
}
```

⚠️ **Schema 迁移**：新增 model 属可加变更，**保持单一 `SchemaV1` + 自动轻量迁移**。**禁止**误加 `SchemaV2` 仍指向同一 live 类（会触发 "Duplicate version checksums" 启动崩溃，见 `dateless-planning-trips.md` 同款坑）。

### 上下文粒度（`contextKey`）—— 关键决策点
样本稀疏是最大约束（单用户一年没几趟）。粒度越细越学不动。建议：
- **主：`scene:<sceneKey>`**（如 `scene:tropical`）—— 最稳，跨目的地聚合同类场景。
- **辅(可选，Phase 3 再上)：`country:<code>`**（如 `country:JP`）—— 捕捉"去日本必带暖宝宝"这种与场景无关的目的地习惯。
- **不做**：`scene×city` 这类过细组合，样本永远不够。

## 信号采集

挂在 `TripStore` 现有的增删改节点（即 CLAUDE.md 要求必须维护的那些函数），**采集与推荐解耦**——Phase 1 只记不改。

| 信号 | 采集时机 | 计数 |
|---|---|---|
| 推荐项被保留 | 行程"完成/离开编辑"时，推荐过且仍在清单 | `keptCount += 1` |
| 推荐项被删除 | 用户删掉一个本由场景推荐的物品 | `removedCount += 1` |
| 手动新增非推荐项 | 用户加了一个当前场景未推荐的物品 | `manualAddCount += 1` |

- 归因：以该行程的 `selectedSceneKeys`（已有字段）+ 目的地 `countryCode` 作为 `contextKey` 来源。
- 需要一个"本次推荐了哪些项"的基线快照，才能区分"删除推荐项" vs "删除自己加的项"。

## 生成时调整逻辑

在现有 `generatePackingSections(...)` 输出后叠一层 `personalize(...)`：

1. **降权 / 移除噪音**（最安全，Phase 2 先做）：
   `removedCount ≥ 2 且 keptCount == 0` → 该项不再自动加入清单（可降级到"顺手考虑一下"，不直接消失）。
2. **升权 / 学习惯**（Phase 3）：
   `manualAddCount ≥ 2` → 该项升入推荐，或先在"顺手考虑一下"露出。
3. **保守阈值**：任何调整至少需 **2~3 次**观察；样本不足时完全回退到静态表。
4. **核心白名单**：`baseItems` 及关键证件类永不被降权移除。
5. **时间衰减**：`lastSeen` 超过 ~12 个月的计数按比例衰减或忽略，让"戒掉的习惯"被遗忘。

## UI / 露出

- **默认隐形**：个性化后的清单看起来就是更准了，无新增 UI、无 toast、无标记。
- **可控露出**：被升权/将被降权的项，优先进**已有的"顺手考虑一下"**区，由用户决定，而非强行增删。
- **不做**：成就感式"我们已学习你的 N 次出行"等张扬提示（违背克制原则）。
- （待议）设置内是否提供"重置我的打包习惯"开关，给用户兜底掌控感。

## 分阶段计划

| 阶段 | 内容 | 风险 |
|---|---|---|
| **上线前** | **不做**。保持静态 `sceneItemMap`。 | — |
| **Phase 1（上线后）** | 建 `PackingHabit` model + 信号采集，**只记不改行为** | 极低（用户无感，纯攒数据） |
| **Phase 2** | 用数据做**降权**（移除用户明显不要的噪音） | 低（只减不增，最安全） |
| **Phase 3** | **升权** + 引入 `country:` 维度，经"顺手考虑一下"露出 | 中（需防过度个性化） |

## 风险 / 陷阱

1. **过度个性化 → 推荐不可预测**，反伤信任。阈值保守、核心稳定、静态表始终兜底。
2. **样本稀疏**：单用户数据少。坚持 `scene:` 粗粒度；不足就回退默认。
3. **归因依赖"推荐基线快照"**：没有它无法区分删的是推荐项还是自加项，信号会脏。
4. **物品名是存储 key**：重命名物品必须在 `itemNameAliases` 留映射，否则历史习惯断链（CLAUDE.md 硬约定）。
5. **冷启动公平性**：新场景/新目的地永远先用静态表，个性化只在有历史后叠加。
6. **埋点闭环**：若新增 `CarryLogger.Event`，须同次补齐调用点（禁止"先定义后接线"）。

## 待确认决策（评审时拍板）

1. `contextKey` 粒度：仅 `scene:` 起步，还是首发就带 `country:`？（建议仅 scene 起步）
2. 降权后是"完全不加"还是"降到顺手考虑一下"？（建议后者，更温和可逆）
3. 阈值具体取值（kept/removed/manualAdd 各几次触发）——需 Phase 1 数据回校，spec 先占位 2~3。
4. 是否提供"重置打包习惯"设置开关。
5. 时间衰减窗口（建议 ~12 个月）与衰减方式（硬截断 vs 加权）。
