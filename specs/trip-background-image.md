# Trip Background Image — Spec

> **Status: Draft（待确认，未实现）** — 2026-06-04
> 关联：首页改版（`feature/home-ui-redesign` 分支）已做好"卫星图大卡"作为默认视觉，本 spec 在其之上增加"用户自定义背景图"。

## 背景 / 目标

首页行程卡从纯文字升级为"有地点感的画面卡"后，视觉冲击力来自背景图像。本功能让每个行程拥有一张背景图，并允许用户自定义，对标 Tripsy。

核心约束（决定了方案选型）：
- **中国区上架 + 主力用户在大陆**：不能强依赖在大陆不稳定的境外图库。
- **独立开发**：不引入需要服务器/CDN/ICP 备案的重基建（至少阶段一不引入）。
- **任何情况下卡片都不能空**。

## 三级兜底（核心设计）

每个行程的背景图按优先级取，保证永远有画面：

1. **用户自定义图**（本地相册上传 / 在线图库选图）— 最佳。
2. **目的地卫星图**（MapKit 快照，已实现 `DestinationMapImage`）— 零操作兜底，大陆/全球都稳。
3. **渐变占位**（无坐标时）— 最末兜底。

> 关键：境外图库（阶段二）只是"能联网时的便利"，失败时优雅降级到卫星/本地，不影响主体验。

## 用户流程

1. **创建行程**：输入目的地城市后，新增入口「选择背景」（可跳过，跳过即用卫星兜底）。
   - 「从相册上传」→ `PHPicker`（无需相册权限弹窗）。
   - 「在线搜索」（阶段二）→ 用城市名查图库 → 网格展示 → 选一张。
2. **编辑行程**：`EditTripView` 同样可改背景图（或移除，回退到卫星）。
3. 选中后：图片**下载/压缩后存本地**，行程引用它。

## 数据模型 + 迁移

- 给行程模型新增背景图引用字段（需求评估两种存法，**建议存本地文件名/相对路径，不存 URL**）：
  - 理由：① 本地上传本就必须存本地；② 离线可用；③ 绕开大陆网络对境外 URL 的不稳定；④ 统一两种来源。
- 图片文件存 App 沙盒（如 `Application Support/TripBackgrounds/<uuid>.jpg`），**压缩**到卡片所需尺寸（避免原图撑大体积）；可另存一张详情页大图尺寸。
- **SwiftData 迁移**：新增字段属 schema 变更，必须走 `CarrySchema` 版本化 + `CarryMigrationPlan`（不得裸改 model）。
- 记录来源元数据：来源类型（local / unsplash / pexels）、（若境外库）摄影师 + 链接（**署名合规**用）、download_location（合规追踪用）。

## 备份策略

- 本地上传的图**无法重新拉取**，必须纳入备份还原（`CarryBackup` / `DataBackupManager`）。
- 方案：备份时把背景图（压缩版）随数据一起导出；还原时落回沙盒。
- 注意备份体积：每图限制尺寸/质量；评估上限（如单图 ≤ 200–300KB）。

## 图源方案

### 阶段一（零外部风险，先落地）
- **本地相册上传**（PHPicker）：全球 + 大陆 100% 可用、离线、零成本、零版权风险。
- **卫星图默认**：已实现。
- 不接任何在线图库 → 无网络依赖、无合规负担。

### 阶段二（可选，联网便利，验证后再上）
- **境外图库（Unsplash / Pexels）实时搜索**，失败优雅降级。
- ⚠️ **大陆访问稳定性待实测**：Unsplash/Pexels 的 API + 图片 CDN 在大陆"可达但不稳"，上线前必须真机/真网实测；不稳则不做或转代理方案。
- **Unsplash API 合规清单**（不遵守会被封 key）：
  - [ ] 显示**署名**（摄影师名 + Unsplash，带链接，含 UTM）。
  - [ ] 用户选图时触发其 **download_location** 接口（下载统计）。
  - [ ] 申请 **production key**（demo 限 50 次/小时，正式 5000/小时）。
  - [ ] 不复刻其核心浏览体验（"选背景"合规）。
  - [ ] Pexels 若用，单独核对其条款（署名建议、限流）。
- **"大陆也稳的实时搜索"正解**（成本高，暂不做）：境外库取图 + 图片走自建大陆可达 CDN（阿里/腾讯云 + 域名 ICP 备案）中转。需服务器 + CDN + 备案，独立开发负担重，**明确暂缓**。

### 不采用
- 国内图库（视觉中国/图虫/海洛等）：大陆稳但海外慢、且**付费 + 版权严苛**（法务风险），否决。
- 自带精选图包：可作为未来增强（覆盖热门城市、离线全球可用），本期暂不纳入，先看本地上传 + 卫星是否够。

## 性能 / 缓存

- 卫星图已有进程内 `NSCache`（`DestinationMapCache`）。
- 用户图：本地文件 + 内存缓存；列表滚动复用，避免每帧重读/解码大图（按显示尺寸解码）。

## 合规

- **隐私**：在线搜索会把"城市名"发给第三方图库（非个人数据），需在隐私说明体现网络调用；本地上传不涉及。
- **大陆**：阶段一无境外网络调用，最稳；阶段二须先验证访问 + 合规署名。
- **App Store 审核**：第三方图内容需注意不当图过滤（图库一般已过滤，但选图网格建议有举报/兜底）。

## 分阶段落地计划

1. **阶段一**：数据模型 + 迁移 + 沙盒存储 + 备份 + PHPicker 本地上传 + 卡片/详情页用图（无图回退卫星）。→ 零外部依赖即可上线。
2. **阶段二（待定）**：Unsplash/Pexels 在线搜索 + 合规署名/追踪 + 大陆降级。先做大陆访问调研。

## 已确认决策（2026-06-04）

1. **阶段一只做"本地上传 + 卫星"**，不做在线搜索。但**阶段一的数据模型与交互框架必须为阶段二预留**，避免日后大改（见下「可延展设计」）。
2. 背景图**存本地压缩副本**（无服务器）。
3. **阶段一一张背景图**，但数据结构按"可多张"设计（多目的地未来可扩展，不改 schema / 不大改 UI）。
4. 背景图**详情页顶部也用大图**（体验更统一）。
5. **无用户可见上限**；内部按显示尺寸压缩（手机原图 3–8MB → 压到 ~150–400KB，视觉无损）。理由：避免 App 数据 / 备份膨胀、滚动解码卡顿。压缩目标内部定（JPEG ~0.7、长边 ~1200px 量级）。
6. **精选图包暂缓**（卫星 + 本地上传先够；未来增强）。

## 可延展设计（保证阶段二 / 多张不大改）

**数据（来源无关、数组化）**：行程新增 `backgroundsData: Data`，编码一个 `[TripBackgroundEntry]`（Codable），套用现有 `additionalDestinationsData` 同款做法。每条：
```
TripBackgroundEntry {
    source: "satellite" | "local" | "unsplash" | "pexels"
    localFileName: String?      // 沙盒文件名（local / 已下载的在线图）
    destinationIndex: Int       // 对应第几个目的地（阶段一恒为 0）
    // 在线来源专用（阶段一为空，阶段二填上即可，模型零改动）：
    photographerName: String?
    attributionURL: String?     // 署名链接
    downloadLocationURL: String? // Unsplash 下载追踪
}
```
- 阶段一只放 1 条（destinationIndex 0）；多目的地未来**往数组追加**，UI 循环渲染。
- 在线来源字段已预留 → 阶段二接 Unsplash 仅"填字段"，**不改 schema**。
- SwiftData：加 `backgroundsData` 属于 schema 变更，走 `CarrySchema` 版本化 + 轻量迁移（带默认值 `Data()`）。

**交互（来源无关的选择器）**：「选择背景」做成一个 sheet：
- 阶段一：`[从相册]`（PHPicker）+ `[使用卫星图(默认)]` + 移除。
- 阶段二：**加一个 `[在线搜索]` tab**，sheet 框架不动。

**取图解析器**：卡片 / 详情统一调一个 resolver，按三级兜底返回图：用户图 → 卫星(`DestinationMapImage`) → 渐变。

**存储**：`BackgroundImageStore`（沙盒 `Application Support/TripBackgrounds/<uuid>.jpg`，写入即压缩）。纳入 `DataBackupManager`（本地上传图无法重拉，必须备份）。

## 暂不处理
- 自建图片代理 / CDN / ICP 备案方案（阶段二的"大陆稳定实时搜索"正解，成本高，暂缓）。
- 国内付费图库接入。
- 精选目的地图包。
