# App Navigation Framework（产品主框架交互）

> **Status: Implemented（已实现并模拟器验证；两项有意延后见下）** — 本 spec 定义 Carry 的根级导航与行程内导航框架。后续功能都基于这份框架展开。参考 Tripsy，但有意识地不照抄会损害「高频行程规划即时可达」与「克制」的部分。
>
> **实现与实跑结论（2026-06-12）：** 根级去 TabView、设置→右上 sheet（带 Done）、创建→右下悬浮 FAB；行程内顶部 Segmented → 底部胶囊切换；默认面规则（新建→打包/已有→行程）+ 记住上次面（已验证记忆生效）；「…」trip 动作两面常驻，打包专属动作仅打包面。`detailFaceSwitched` 埋点。模拟器逐项验证通过。
>
> **两项有意延后（诚实记录）：**
> 1. **首页搜索**：原计划「原生下拉 `.searchable`」——但 iPhone 首页是「足迹地球 + UIKit 悬浮 Sheet（`CarryBottomSheetFX`），无导航栏」，原生 pull-down search 需要承载它的 nav bar，这里没有，故**原生方案不适用**。需做**自定义 in-sheet 搜索**（Sheet 头部一个搜索入口/可展开搜索框）——属独立增量，且触碰性能敏感的 FX Sheet，单独排期，不塞进本框架。当前 Carry 本就无搜索，属纯增量、不阻塞框架。
> 2. **行程内底部「+」上下文添加**：原 Tripsy 参考里有；但 Carry 两面**都已有清晰的内联添加**（打包：分区内「+ 添加物品」/「…」从库添加；行程：内联「加一天 / 加停靠点」）。再加一个全局「+」会**语义含糊（加什么？）且不克制**，故不做。如日后需要再评估。

## 动机

旧结构两处别扭：
1. **根级用 TabView（行程 / 设置）**：设置是最低频的东西，却占一个常驻底部 Tab，还把首页劈成 TabView。
2. **行程内用顶部 Segmented（打包清单 ｜ 行程规划）**：把两个**生命周期/频次完全不同**的面当对等平级——打包=出发前一次性配置；行程规划=旅途中高频反复开（基本每到一个地点就看）。结果是高频视图被放到最难单手够到的顶部，且默认停在低频的打包面。三重别扭，且观感「程序化」。

核心洞察：**「打包」和「行程规划」不是对等的两个 Tab。** 框架要让高频的行程规划默认、易达；让切换落到拇指区。

## 已锁定决策

### 根级（App 启动 = 行程列表）
- **取消底部 TabView。** 根仍是现有 `HomeView`，不再有底部 Tab 栏。
- **⚠️ 首页内容原样保留，本框架只改导航 chrome：** `HomeView` 现有的 **TRIP OVERVIEW 概览（全部行程 / 即将出发 / 到访国家数）、足迹地图、以及 Hero / 即将出发 / 规划中 / 往年 分组列表**全部不动。框架仅调整：底部 Tab 栏（移除）、设置入口（→右上）、创建（→右下+）、搜索（→下拉）。
- **设置 → 右上角头像/入口后面**（低频，不占常驻位）。以 sheet 形式从头像打开。
- **创建行程 → 右下角悬浮「+」**（拇指可达）。
- **搜索 → iOS 原生下拉搜索（`.searchable`）**，不占常驻位，不重建底部搜索栏。
- **不做**「中间行程簿切换器」（Tripsy 的 My Tripsy Book）——Carry 暂无多簿概念，那是冗余。留作未来「行程簿/文件夹」的伏笔，届时再加中间切换器。

### 行程内（进入某个行程）
- **底部悬浮胶囊 Segmented：行程 ｜ 打包**（拇指可达，单手切换）。样式见 Mockup：居中胶囊浮在底部，不挡内容，区别于通用 Tab 栏的「程序化」。
- **默认面规则**：
  - **新建的行程 → 默认「打包」**（刚建完就是配置装备清单，贴合 `isNewTrip` 现有落点）。
  - **已有行程 → 默认「行程规划」**（旅途中高频）。
  - **记住每个行程上次看的面**：再次进入时优先回到上次停留的面；无记录时按上面的新/旧规则。
- **行程动作「…」→ 右上角**：改信息/日期、换封面、提醒、分享、删除等 trip 级动作。两个面下都常驻（trip 级，与当前面无关）。
- **「+」→ 右下角，按当前面上下文**：行程面 = 加天/加停靠点；打包面 = 加物品。
- 打包面**特有**的动作（从物品库添加、标记完成、编辑分类）仍归属打包面（可放该面内或「…」分组），不混入 trip 级「…」。

### 设置
- 内容不变，仅从底部 Tab 迁到根列表右上角入口（sheet）。

## 有意不照抄 Tripsy（护住核心诉求）

- **「行程作为悬浮 Sheet 盖在地图上」**：大改；且打包清单盖在地图上无意义；Carry 已有行程内地图块 + 足迹地球。留作未来「wow」，不卡框架。
- **「Hub + 卡片（行程/费用/文件/航班…点进去）」**：会在行程规划**上面再加一层** Hub，使最高频的视图反而深一级，与「每到一地秒开行程规划」相悖。Carry 用「双面 + 底部切换」更直接。费用/文件等以后想要，可做成第三个面或塞进「…」，不动框架。

## 实现要点（待开发）

- **`ContentView`**：移除 `TabView`（Trips / Settings）。根 = `HomeView`（行程列表）。`NavigationRouter` 路由保留；设置改由 `HomeView` 右上入口以 `.sheet` 呈现。
- **`HomeView`**：右上角加头像/设置入口；`.searchable` 下拉搜索；右下角悬浮创建「+」（复用现有创建流程入口 `CreationRoute.tripInfo`）。
- **`PackingListView`（行程详情）**：
  - 移除顶部 Segmented（Phase 2 加的 `DetailTab` + 顶部 Picker）。
  - 改为**底部悬浮胶囊切换** + 右下「+」（上下文化）+ 右上「…」（trip 动作，两面常驻）。
  - **默认面逻辑**：`isNewTrip → .packing`；否则读「上次看的面」（无则 `.itinerary`）。
  - **记住上次面**：按 trip id 存 `UserDefaults`（轻量；键如 `itinerary_last_face_<uuid>`），切换时写入。
- **Siri / Quick Actions / 深链**：`open_trip` 打开行程后应落到「默认面规则」；`show_map`/Footprint 仍走足迹地球（独立）。检查 `CarryShortcuts` 无对「打包 Tab」的硬假设。
- **埋点**：面切换可加 `detailFaceSwitched`（评估两面频次，验证「行程规划更高频」的假设）；定义即接线。
- **本地化**：胶囊文案复用 `detail.tab.packing` / `detail.tab.itinerary`（已存在）；设置入口、搜索 placeholder 等新文案补 9 语言。

## 不在本框架范围内

- 多行程簿/文件夹（未来再加根级中间切换器）。
- Sheet-over-map 的行程呈现、Hub 卡片式行程主页。
- 费用/文件作为独立第三面（可后续评估）。

## 建议实现顺序

1. 根级去 TabView：`HomeView` + 右上设置 sheet + 右下创建 +（先把根跑通）。
2. 行程内：移除顶部 Segmented → 底部胶囊切换 + 默认面规则 + 记住上次面。
3. 「…」(trip 动作) 与 「+」(上下文添加) 落位；打包特有动作归位。
4. 原生下拉搜索 + 本地化 + 埋点。
5. 模拟器实跑：新建行程默认打包；已有默认行程规划；切换记忆；设置/创建/搜索可达；单手够到底部切换。
